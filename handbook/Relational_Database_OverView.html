    RDBMS Architecture Overview

    # RDBMS Architecture Overview

    Five detailed levels, in my view, Buffer Pool Manager + Disk Manger -> Storage Engine, Operator Execution + Access Methods -> Execution Engine, Query Planning -> SQL Parser Engine

    # Query Planning

    Parser (output: abstract syntax tree)

    What is abstract syntax tree?

    生成的语法树就是一个保留原语句的结构化的树。比如，把下面示例语句编译成语法树

    SELECT
classrCOUNT(id) AS count,
MAX(id) AS max id
MIN(id) AS min id
FROM
SELECT
class,
id
FROM
studentWHERE
gender ='female
GROUP BY
class;

    语法树be like:

    值得注意的是，parser只能检查语法和词法，比如SELECT拼写，多余的标点符号。

    Binder (output: bounded syntax tree -> logical plan)

    首先两步走，第一步生成受限语法树，syntax没法检查出语义错误，比如select from不存在的表，不存在的属性，错误的表达式类型等等，这就需要binder去把syntax中的关键字和数据库的元数据（metadata）进行绑定，从而赋予语义（semantic）。第二步，binder用关系代数运算符去替换syntax tree上的节点和结构，生成逻辑查询计划，也就是关系代数表达式。

    Tree Rewiter (output: optimized logical plan)

    1. 模式匹配来去掉或者合并一些condition。
    where val between 1 and 100 or val between 50 and 150;
->
where val between 1 and 150;
    1. 利用代数定律优化逻辑查询计划，获得逻辑计划2.0. 以此sql为例，展示四个最常见的优化方法：
    SELECT ARTIST.NAME
FROM ARTIST, APPEARS, ALBUM
WHERE ARTIST.ID=APPEARS.ARTIST_ID
AND APPEARS.ALBUM_ID=ALBUM.ID
AND ALBUM.NAME="Andy's OG Remix"

    1. Split Conjunctive Predicates 条件分解
    2. Predicate Pushdown 选择下推
    3. Replace Cartesian Products with Joins 积运算换成等值连接
    4. Projection Pushdown 投影下推

    逻辑计划生成参考 (opens new window)： https://www.cnblogs.com/jason-t/p/16113027.html

    Optimizer (output: physical plan) 这一部分由逻辑计划派生得到多个物理计划，如何对多个物理计划进行评估选择，是optimizer需要做的。 两类优化方法，通常结合起来

    1. Heuristics / Rules 启发式优化/固定规则； 上面的逻辑计划优化中就使用的启发式规则，说白了就是静态固定的优化逻辑。在物理计划的优化中同样可以用启发式方法来选择更优的物理计划。
    2. Cost Model and Cost-based Optimization 基于代价模型的优化 两个部分：如何建立代价模型； 如何进行选择。

    # Operator Execution

    Execution Model:

    1. Iterator Model Also called Volcano / Pipeline Model 迭代/火山/流水线模型

    每个操作符都实现Next()方法，父节点通过for循环调用子节点迭代返回元组。有些操作符比如join, subqueries, order by例外，需要阻塞到子节点返回全部元组。最常用的模型

    1. Materialization Model

    每个操作符一次性全输入，一次性全输出。适合OLTP,小数据量，可以较少调用的overhead。

    1. Vectorized / Batch Model

    批量输入，批量输出。每次next()返回的是多行元组。利用CPU的SIMD向量化特性，也就是Spatial Locality的特性。OLAP最爱。计算密集型。所以向量化的缺点是什么？可能的答案：火山模型的输出控制更容易，对于OLTP数据库向量化带来的提升有限？

    1. 向量化+编译执行 貌似是比较新的优化执行方法，mark这篇文章 (opens new window)。
